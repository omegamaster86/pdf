# How I Use Cursor（和訳）

**出典:** [lauren (@poteto) on X](https://x.com/poteto/status/2058975157503570132)  
**著者:** lauren（Cursor ソフトウェアエンジニア）  
**公開日:** 2026年5月25日

---

## はじめに

正直に言わせてほしい。@cursor_ai の面接の前まで、私は実際には Cursor を使ったことがなかった。

Meta では Claude Code が爆発的に広がっていた。サイドプロジェクト用に月額200ドルの個人プランまで払っていた。シンプルさと、すぐに生産的になれる速さが好きだった。ただ、自分なりのスキルセットを育てて cc をほぼ何にでも変えられるようにするあたりで、私にとっての引っかかりどころだった。cc の上に自分専用のエージェントオーケストレーターツールまで作り始めていた。

## 面接で Cursor を使ってみた

オンサイト面接では2日間 Cursor を使い、面接プロジェクトを組み立てた。これは Cursor 3 のリリース前なので、Editor Window を使っていた。ここ数年ずっと vscode を使ってきたおかげで、キーボードショートカットの多くはまだコンテキストに残っていた。IDE に戻るのはそれほど難しくなかった。ただ、正直に言うと最初の1〜2時間は確かに CLI が恋しかった。クリック操作はほとんど野蛮に感じた。それでも、いくつか本当に目立った点があった。

**第一に**、当時使っていたモデル——Opus と Codex——は、なんとなく賢く感じた。プロジェクトの異なる部分でモデルをその場で切り替え、両方を同時に使えるのは素晴らしかった（フロントエンドは Opus、システムは Codex）。面接前からマルチモデルによる対立的レビューを絶賛していたので、UI 上でネイティブにそれができるのはとても自然に感じた。さらに良かったのは、異なるモデルのサブエージェントを起動できること。1つの会話の中で両方の良いところを取れる。

**第二に**、コンパクションが異常なほど速かった。cc ユーザーとして、コンパクションに何分もかかるのに慣れていたので、常にコンテキストとプラン使用量を警戒していた。Cursor ではどれだけ速いか、完全に驚いた。もうコンテキスト使用量をほとんど気にする必要がなかった。cc ではコンパクション後にモデルがすごくバカになったように感じることが多かったのに対し、Cursor ではただ動いてくれた。

**第三に**、GUI が TUI よりどれだけ多くのことを提供できるかに気づいた。Cursor のブラウザでアプリを直接開き、Design Mode でデザイン変更できるのは直感的だった。用途特化型 UI がエージェント型コーディングをどれだけ効果的にできるか、考えさせられた。

## Cursor で Cursor を作る

3月末に入社してから、主に Cursor 3 の Agent Window に取り組み、それを日常使いにしている。cc はクールなプロダクトで素晴らしいチームだと思うが、そのシンプルさは人々を cc を包む独自の抽象化を作りたくさせる傾向がある。前職では、cc の上に作られた新しい社内オーケストレーターツールが毎週発表されているように感じた。

@bcherny は「潜在需要（latent demand）」という考え方をよく語る：

> 「プロダクトにおけるとても古い考え方で、潜在需要というものがある……ハッカブルで、他のユースケースに流用できるほどオープンエンドにプロダクトを作る。そして人々がどう流用するかを見て、それに向けて作る。」

まさにそれだった。オーケストレーションツールに収束する人々は、CLI を使うと人間自身がオーケストレーターになるという潜在需要を露わにしている。

しかし、私が使ってきたエージェントワークフローは間違ったことに焦点を当てていた。GUI で複数の CLI を動かすことは、要点を完全に外していた。私が関心を持っていたアプローチは、エージェントへの信頼を築くことだった。

元エンジニアリングマネージャーとして、エージェントの管理は人間のエンジニアリングチームを組むことに似ているとすぐ気づいた。新入社員はコードベースを理解するだけでなく、仕事の進め方も理解するためのオンボーディングが必要だ。彼らは過去の経験から得たスキル——デバッグの仕方、高品質なコードとテストの書き方、コミュニケーションの仕方など——をすでに持って入ってくる。

エージェントは、常に健忘症と愚かさの状態にある新入社員のようなものだ。言ったことを覚えておらず、本当に新しいことを学ぶこともない。ただし、ルール、スキル、ツール、長期記憶を装備させることで、それに近づける。有能だが愚かで、とても教えやすい。彼らの失敗モードを、深く厳密なエンジニアリングについて私が知っていることをすべて教える機会だと見なした。

厳密さがないと、エージェントはお世辞を言いながら、頼まれたコードを書くために何でもする。そして、大量に書く。素朴な並列化は、ただスロップをより速く書かせるだけだ。

> 多くのエージェントを並列にオーケストレーションする価値は、広くではなく深く行くことから来ると、ますます確信している。単一または少数の問題についてより深く掘り下げ、最高の結果を得る確率を最大化したい——ベストオブNスタイルのレースで最良の解を見つける、など。

## 速く行きたければ、まず深く

エージェントのオーケストレーションは生産的にできると思う。ただし、深さ優先でなければならない。

私は pstack——@cursor_ai を毎日構築するために使っている個人のスキルとエンジニアリング原則のセット——をオープンソースにする。これらのスキルの初期版はサイドプロジェクトで作り始め、それ以来ずっと洗練してきた。

**入手先:** https://cursor.com/marketplace/cursor/pstack

```
/add-plugin pstack
```

これらのスキルは Cursor チーム内でも最も使われているスキルのひとつになったので、みなさんにも共有できるのが楽しみだ。

pstack は複数モデルを使ってエージェントにより厳密にさせる。観察してきた失敗モードをすべてスキルに落とし込んだ。プラグインの中核は `/poteto-mode` で、与えられたタスクに対してエージェントが従うべき正しいプレイブックを与える高階スキルだ。目標は LOC の最大化ではなく、その逆——最小限のコードで最大のインパクトだ。

厳密さは、経験豊富なエンジニアと同じやり方で問題にアプローチすることで適用される。たとえば、デバッグへの優れたアプローチは問題空間を二分探索することだ。何が起きているかについての仮説から始め、体系的に除外していき、真の根本原因に近づく。再現が難しければ、バグを人工的に発生させることを試すかもしれない。計装やコンソールログを追加して、実行中のプログラム状態を確認することもある。

これらのステップは、推測するのではなく徹底的に問題をデバッグするためにエージェントが使えるプレイブックを形成する。放置すれば喜んで推測するエージェントに対してだ。pstack には、同じレベルの厳密さでソフトウェアエンジニアリングにアプローチできる多くのスキルとプレイブックが同梱されている。現在のプレイブック：

- スキル作成と eval
- 自律的な作業
- バグ修正とランタイムフォレンジック
- 機能開発
- ビジュアルパリティとプロトタイピング
- その他

厳密さが必要なときは、プロンプトの先頭に `/poteto-mode` を付ける。例：

```
/poteto-mode this pr has a subtle bug where the scroll drifts every 750ms even when idle. repro first, then fix and verify.
/poteto-mode a big list takes a second or two to load even though we virtualize. run a cpu trace and tell me why.
/poteto-mode build a small feature behind a feature flag. verify it really works.
/poteto-mode build two prototypes of the markdown renderer so we can compare. spawn an agent for each.
/poteto-mode open source these skills as a plugin. nothing internal leaks, work in a temp dir, show me the dependency graph first.
/poteto-mode i'm going to bed. land the stack even if ci flakes. i want everything merged by morning.
/poteto-mode the row spacing is too tall when this flag is on. the second image is correct. repro and fix until it matches.
```

```
/poteto-mode この PR には、アイドル時でも750msごとにスクロールがずれる微妙なバグがある。まず再現してから修正し、検証して。
/poteto-mode 仮想化しているのに大きなリストの読み込みに1〜2秒かかる。CPU トレースを実行して理由を教えて。
/poteto-mode フィーチャーフラグの裏に小さな機能を作って。本当に動くか検証して。
/poteto-mode 比較できるように markdown レンダラーの2つのプロトタイプを作って。それぞれにエージェントを起動して。
/poteto-mode これらのスキルをプラグインとしてオープンソース化して。内部情報は漏らさず、一時ディレクトリで作業して、まず依存関係グラフを見せて。
/poteto-mode 寝るから。CI がフレークしてもスタックを着地させて。朝までに全部マージしてほしい。
/poteto-mode このフラグがオンのとき行間が高すぎる。2枚目の画像が正しい。一致するまで再現して修正して。
```

オンデマンドで他のスキルも呼び出せる：

- **/how** — サブシステムが実際にどう動くかのウォークスルーが欲しいとき
- **/why** — なぜこのように作られたか知りたいとき。利用可能な MCP で各証拠カテゴリ（ソース管理、課題トラッカー、長文ドキュメント、リアルタイムチャット、インフラ可観測性、エラー追跡、アナリティクスウェアハウス）を並列クエリする
- **/architect** — 関数境界をまたぐコードを書く前に、型とデータ構造を先に固めたいとき
- **/arena** — 同じことに N 回並列で挑戦し、それぞれの良い部分を取り込みたいとき
- **/interrogate** — 異なるモデルが対立的にレビューしてほしいとき
- **/tdd** — バグを直すとき。まず失敗するテストを書き、その後修正する
- **/unslop** — AI 臭のある文章を整えたいとき。平易に書かせる
- **/reflect** — 長い会話の後にスキルを継続的に改善したいとき
- **/figure-it-out** — 珍しいことをしているとき。厳密で監査可能なプレイブックを設計する
- **/show-me-your-work** — レビュー可能な意思決定の痕跡が欲しいとき。決定をコミットできる TSV に記録する

最後に、**/automate-me** で自分専用のモードスキルを作れる。最近のトランスクリプトをマイニングし、あなたの働き方から your-mode スキルを下書きし、下で pstack にルーティングする。

pstack はどのエージェント型コーディングツールでも動くが、Cursor のようなマルチモデルツールでは特にうまく機能する。多くのスキルは、各モデルの独自の強みと弱みを活かすマルチモデルワークフローを使う。幅優先ではなく深さ優先に適用されたエージェントオーケストレーションだ。

エージェントのボトルネックは検証にある。エージェントは大量のコードを素早く書ける。すべてが正しいことを確認するのは極めて難しい。そこに到達できれば、ソフトウェアのダークファクトリーのような真のエージェント並列化が可能になるかもしれない。

ただし、まず深く、厳密でなければならない。信頼を高めていくことでそこに到達できると思う。

pstack を試して、感想を聞かせてほしい。

## ソフトウェア保守の禅と技術

これらのスキルは、コードを書くときにより自信を持って動けるようにしてくれる。ただ、エージェントがすべてのコードを書くようになってから、保守は悪夢になった。バグ、パフォーマンス問題、機能リクエストの対応には依然として時間がかかる。そして量がずっと増えた。

Cursor では Cursor のオートメーションを多用している。スケジュール実行や、Slack チャンネルの新着メッセージなどのイベントに応じて動くクラウドエージェントだ。一例がボットの Benny。pstack と同じスキルを彼に与えている。

Benny はまだ作業中だが、ソフトウェア保守プロセスをできる限り自動化するのがビジョンだ。考え方はこうだ：pstack で問題をほぼ「ワンショット」でき、PR の品質が高いとある程度確信できるなら、フィードバックも自動化できるはずだ。

このファクトリーはトリアージから始まる：従業員からのバグ報告情報の収集。Cursor をたくさんドッグフーディングしているので、リリース候補に対する従業員からのフィードバックが大量にある。Benny は画像と動画添付を理解し、pstack スキルでコードベースを探索し、再現手順が不明なら報告者とチャットして情報を集める。

これはバグ報告プロセスの重要な部分だ。明確な再現手順と何が壊れているかの理解がなければ、エージェントは解決策を推測するしかない。どこでどう壊れるかを正確に理解させる必要がある。

トリアージ後、Benny はコード調査、最近のバグ回帰の git 履歴、同じバグについての Slack メッセージ、機能がどう動くべきかのデザイン・プロダクト判断の Notion まで見た結果をまとめたチケットを作成する：バグなのか、そう設計されたのか？

チケットが作成されると、別の Benny ボットが **/orchestrate** というスキルで拾う。

まず、コンピュータ操作で問題の再現を試みる。Cursor Cloud Agents はクラウド上で Cursor 自体を実行し、デスクトップと対話し、クリックやキーボード入力を送れる。社内では CDP などのプロトコルでプロダクトをプログラム的に制御するスキルも使っている。

これにより、バグ報告が再現できるか示せる。一貫して再現できれば修正を試みる。パフォーマンス問題なら、修正前後の CPU トレースとヒープスナップショットを取れる。サブプランナーがさらにワーカーを起動し、pstack スキルで修正を検証し、チケットに対して直ったか確認する。

この実行では、修正前後の動画を撮るワーカーや、説明に動画を載せて PR を開くワーカーも追加で起動される。

成功裏に再現・修正された最近の実行例もある。

すべてまだ作業中で、やることは山ほどあるが、寝ている間や他のことをしている間に、自信を持ってバグを直してくれるエージェントチームがあることにワクワクしている。コードレビューのスケーラビリティも大きな領域で、Cursor には役立つ機能が近いうちに出てくると思う。

ただし、自分のソフトウェアファクトリーを作る鍵は信頼だ。検証を含めてエンドツーエンドで問題を任せられるエージェントを信頼できなければ、プロセスを自動化できない。pstack のようなプラグインでエージェントにエンジニアリングの深さを与え、信頼を高めていけば、より野心的な問題に取り組み始められる。まだ信頼していないエージェントを並列化しようとするのは、トークンの巨大な無駄遣いで、コードベースにもっとスロップを入れる。

読んでくれてありがとう！
